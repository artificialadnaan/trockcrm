Request Flow — Schema-Per-Office Routing

1. Browser → POST /api/deals (Authorization: Bearer {jwt})
2. Auth middleware → validates JWT, extracts { userId, officeId, role }
3. Office resolver → looks up office.slug from officeId → "dallas"
4. Schema setter → SET LOCAL search_path = 'office_dallas', 'public'
5. Audit setter → SET LOCAL app.current_user_id = '{userId}'
6. Route handler → db.insert(deals).values({...}) — hits office_dallas.deals
7. PG trigger → writes to office_dallas.audit_log automatically
8. Event bus → emits deal.created → listeners fire (touchpoint alert, notification)
9. SSE → pushes notification to assigned rep's browser

Architecture Decisions Summary

  • Schema routing via search_path: One middleware line sets the schema for the entire request. Drizzle queries don't need office_id filters — PG resolves the correct table automatically.
  • Cross-office reporting: Director/admin queries use UNION ALL across schemas or a materialized reporting view refreshed by the worker.
  • Event bus stays in-process: Node EventEmitter for Phase 1. If scaling requires it, swap to PG LISTEN/NOTIFY or a message queue later — event handler signatures don't change.
  • SyncHub integration: SyncHub feeds Bid Board opportunities to the CRM via internal API or shared database read. CRM owns deal lifecycle from that point forward.
  • Worker shares codebase: Same TypeScript, same Drizzle schemas, different entrypoint. No code duplication.
