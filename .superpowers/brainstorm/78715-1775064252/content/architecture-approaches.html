Architecture Approaches

Three options for the T Rock CRM platform architecture — click to select your preference

A

Modular Monolith

Single Express server + Worker, feature modules with clear boundaries

── server/
modules/
deals/ ← routes, service, validation gates
contacts/ ← directory, dedup, touchpoints
email/ ← Graph API, sync, matching
procore/ ← API client, stage mapping, webhooks
photos/ ← R2 presigned URLs, timeline
reports/ ← saved filters, locked reports, charts
audit/ ← PG triggers setup, history queries
office/ ← schema management, tenant routing
tasks/ ← daily list, reminders, alerts
auth/ ← MS Entra SSO, sessions, RBAC
── worker/
jobs/ ← stale deals, email sync, dedup scan, activity alerts
── client/
React + Vite + Tailwind + shadcn/ui
── shared/
schema/ ← Drizzle schemas, types, enums

Pros

  • Simple deployment — one API service, one worker, one frontend
  • Modules share DB connection, auth middleware, logging
  • Fastest to build for April deadline
  • Easy to navigate — feature = folder
  • Same pattern as SyncHub

Cons

  • Modules can leak into each other if boundaries aren't enforced
  • Single server handles all API traffic — fine for T Rock's scale
  • Large server/ directory as features grow
B

Service-Oriented (Microservices Lite)

Separate Express services per domain, shared database

── services/
deals-api/ ← :3001 — deals, pipeline, estimates
contacts-api/ ← :3002 — directory, dedup
comms-api/ ← :3003 — email, notifications
integrations-api/ ← :3004 — Procore, HubSpot
files-api/ ← :3005 — photos, documents, R2
gateway/ ← :3000 — API gateway, auth, routing
── worker/
Same as Option A
── client/
Same as Option A

Pros

  • True domain isolation — services can't accidentally couple
  • Independent scaling per service
  • One service can restart without affecting others

Cons

  • 6+ Railway services = $30-60/mo extra hosting
  • Inter-service communication adds latency and complexity
  • Overkill for <100 users — adds operational overhead
  • Significantly more build time — won't hit April deadline
  • Cross-service transactions are painful (deal + contact + email in one action)
C

Modular Monolith + Domain Events

Option A's structure with an internal event bus for module communication

── server/
modules/ ← Same as Option A
events/ ← Internal event bus (EventEmitter or pg LISTEN/NOTIFY)
deal.stage.changed → triggers email, Procore sync, task creation
contact.created → triggers dedup check, touchpoint alert
email.received → triggers task creation, deal logging
deal.won → triggers Procore project creation
── worker/ & client/ & shared/
Same as Option A

Pros

  • Same simplicity as Option A — single server, same folder structure
  • Event bus decouples modules — deal module doesn't call email module directly
  • Easy to trace "what happens when X" — check event handlers
  • Natural path to microservices later — events become message queue topics
  • Perfect for CRM workflows (deal won → chain of automated actions)

Cons

  • Slightly more upfront design than pure Option A
  • Event ordering and error handling need thought
  • Can be over-engineered if events are used for everything

⟶ Recommendation: Option C

Same build speed as the modular monolith, but the event bus is a natural fit for CRM automation chains. When a deal is won, you need to: create a Procore project, send a notification, generate tasks, update reports. With direct calls, the deal module imports everything. With events, it emits deal.won and each module handles its piece independently. This is also the cleanest migration path to microservices or message queues if T Rock scales beyond 20 offices.
