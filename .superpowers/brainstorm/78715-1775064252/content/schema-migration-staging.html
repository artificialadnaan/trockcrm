Database Schema — Migration Staging & Procore Sync

Staging tables for HubSpot data migration validation + Procore sync state tracking

HubSpot Migration Staging

These live in a dedicated migration schema — completely separate from live data. Dropped after go-live validation is complete.

migration.staged_deals
id UUID PK DEFAULT gen_random_uuid()
hubspot_deal_id VARCHAR(50) UNIQUE NOT NULL
raw_data JSONB NOT NULL -- full HubSpot API response preserved
-- Mapped Fields (transformed to CRM schema shape)
mapped_name VARCHAR(500)
mapped_stage VARCHAR(100)
mapped_rep_email VARCHAR(255)
mapped_amount NUMERIC(14,2)
mapped_close_date DATE
mapped_source VARCHAR(100)
-- Validation State
validation_status ENUM('pending','valid','invalid','needs_review','approved','rejected') DEFAULT 'pending'
validation_errors JSONB DEFAULT '[]' -- [{ "field": "stage", "error": "Unknown stage: Negotiation" }]
validation_warnings JSONB DEFAULT '[]' -- [{ "field": "amount", "warning": "Amount is $0" }]
reviewed_by UUID FK → public.users.id
review_notes TEXT
promoted_at TIMESTAMPTZ -- NULL until promoted to live
promoted_deal_id UUID -- FK to live deal after promotion
created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
migration.staged_contacts
id UUID PK DEFAULT gen_random_uuid()
hubspot_contact_id VARCHAR(50) UNIQUE NOT NULL
raw_data JSONB NOT NULL
-- Mapped Fields
mapped_first_name VARCHAR(255)
mapped_last_name VARCHAR(255)
mapped_email VARCHAR(255)
mapped_phone VARCHAR(20)
mapped_company VARCHAR(500)
mapped_category VARCHAR(100)
-- Dedup Detection
duplicate_of_staged_id UUID FK → staged_contacts.id -- links to first occurrence
duplicate_of_live_id UUID -- if matches existing CRM contact
duplicate_confidence NUMERIC(3,2)
-- Validation State
validation_status ENUM('pending','valid','invalid','duplicate','needs_review','approved','rejected','merged') DEFAULT 'pending'
validation_errors JSONB DEFAULT '[]'
validation_warnings JSONB DEFAULT '[]'
reviewed_by UUID FK → public.users.id
merge_target_id UUID -- which contact to merge into
promoted_at TIMESTAMPTZ
promoted_contact_id UUID
created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
migration.staged_activities

migration.import_runs
-- Tracks each migration run for audit and rollback
id UUID PK DEFAULT gen_random_uuid()
type ENUM('extract','validate','promote') NOT NULL
status ENUM('running','completed','failed','rolled_back') NOT NULL
stats JSONB NOT NULL -- { total: 412, valid: 389, invalid: 8, needs_review: 15 }
error_log TEXT
run_by UUID FK → public.users.id NOT NULL
started_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
completed_at TIMESTAMPTZ

Procore Sync State

public.procore_sync_state
-- Tracks sync state per entity to prevent conflicts and detect drift
id UUID PK DEFAULT gen_random_uuid()
entity_type ENUM('project','bid','change_order','contact') NOT NULL
procore_id BIGINT NOT NULL
crm_entity_type VARCHAR(50) NOT NULL -- 'deal', 'contact', 'change_order'
crm_entity_id UUID NOT NULL
office_id UUID FK → public.offices.id NOT NULL
sync_direction ENUM('crm_to_procore','procore_to_crm','bidirectional') NOT NULL
last_synced_at TIMESTAMPTZ
last_procore_updated_at TIMESTAMPTZ -- from Procore's updated_at field
last_crm_updated_at TIMESTAMPTZ
sync_status ENUM('synced','pending','conflict','error') DEFAULT 'synced'
conflict_data JSONB -- { field: "status", crm: "In Production", procore: "Active" }
error_message TEXT
created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
UNIQUE(entity_type, procore_id)
INDEX ON (sync_status) WHERE sync_status != 'synced' -- quick find out-of-sync
public.procore_webhook_log
-- Every Procore webhook received, for debugging sync issues
id BIGSERIAL PK
event_type VARCHAR(100) NOT NULL -- "projects.update", "change_orders.create"
resource_id BIGINT NOT NULL
payload JSONB NOT NULL
processed BOOLEAN DEFAULT false
processed_at TIMESTAMPTZ
error_message TEXT
received_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
INDEX ON (processed, received_at) -- unprocessed webhook queue

Migration Workflow & Edge Cases

  • 3-phase migration: Extract (HubSpot API → staging) → Validate (auto-checks + team review) → Promote (approved records → live CRM). Each phase tracked in import_runs.
  • Raw data preserved: raw_data JSONB stores the full HubSpot API response. If mapping logic has a bug, you can re-map without re-extracting.
  • Validation auto-checks: Stage names that don't map to CRM stages, $0 amounts, missing rep assignments, contacts without email or phone, orphan activities with no deal/contact match.
  • Contact dedup at migration: Staged contacts run through the same dedup logic as live contacts. Duplicates flagged with confidence scores for team review. Merge targets selectable in the validation UI.
  • Orphan activities: HubSpot activities that reference deleted deals or contacts get flagged as "orphan" — team decides to discard or manually associate.
  • Rollback support: promoted_deal_id / promoted_contact_id link staged records to their live counterparts. If something goes wrong, you can identify and clean up promoted records.
  • Schema cleanup: After go-live validation is complete, DROP SCHEMA migration CASCADE removes all staging data cleanly.
  • Procore sync conflicts: procore_sync_state detects when both CRM and Procore update the same entity. Conflicts surfaced in admin dashboard for manual resolution — prevents the "Closed Won incident" from SyncHub.
  • Webhook idempotency: procore_webhook_log dedupes by event_type + resource_id within a time window. Procore can send duplicate webhooks.
